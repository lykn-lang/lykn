# DD-50.5 Addendum — Switch Profile Fix and Cross-Compiler Test Coverage

## Why this addendum exists

DD-50.5 (commit `c57a609`) introduced a per-kernel-form context profile in
`crates/lykn-lang/src/emitter/forms.rs` that correctly handles
statement-bodied kernel forms (`while`, `for`, `block`, `try`, etc.) — for
those forms, body children inherit `Statement` context, so a no-else `if`
in the body remains a statement and doesn't trigger DD-50 Rule 2's
compile-error throw.

CDC review on 2026-05-10 verified the profile correctness for most forms
but identified one **entry that is genuinely broken** plus an
**unrelated test-coverage gap that masked the bug**. This addendum
addresses both.

### The bug

The `switch` entry is:

```rust
"switch" => KernelChildProfile::Positional(&[V]),
```

`Positional(&[V])` has a 1-element array. The `context_for` method at
`forms.rs:2546-2555` returns `positions[i]` for `i < array.len()`, else
falls into `unwrap_or(ExprContext::Value)`. So:

- Index 0 (discriminant): `V` ✓
- Index 1+ (case sub-lists): falls through to `Value`. ✗ — should be
  `Statement` per the comment immediately above the entry.

When a switch case body contains a no-else `if`:

```lykn
(switch x (1 (if (= y 0) (console:log "a")) (break)) (default (break)))
```

The case sub-list `(1 (if ...) (break))` is in `Value` context. Its head
`1` is a number (non-atom), so it falls to the line 397-403 "head not
atom" branch which doesn't change context. The case body's `(if (= y 0)
(console:log "a"))` is then processed in `Value` context, hits the
`emit_if_expression` interception at line 375, sees no else, fires Rule 2,
emits a `(throw (new TypeError "COMPILE_ERROR: ..."))` form **replacing
the if**. Compiled JS contains `throw new TypeError(...)` instead of
`if (y === 0) console.log("a")`.

### Why the existing test passes anyway

`test/forms/dd-50_test.lykn` line 129 has:

```lykn
(test
  "DD-50.5: no-else if in switch case body — statement context preserved"
  (bind r (compile "(switch x (1 (if (= y 0) ...))) ...))
  (includes r "if (y === 0)") ...)
```

But `compile` here is from `packages/testing/helpers.js:26`:

```js
export function compile(source) {
  return lykn(source).trim();
}
```

`lykn()` is the **JS compiler** (`packages/lang/mod.js`), NOT the Rust
compiler. The JS compiler has a dedicated `switch` macro handler at
`packages/lang/compiler.js:883-908` that explicitly calls
`toStatement(compileExpr(e))` (line 896) on case body items — no
`'expression'` arg, so the case body's `if` macro receives default
`'statement'` position and produces a normal `IfStatement`. Rule 2 does
not fire on the JS side. The test passes for the right reason on the JS
side and silently misses the broken Rust path.

The Rust unit test for switch (`crates/lykn-lang/src/codegen/emit.rs:2149`
`test_switch`) uses `("a" (f) (break))` — no nested `if` in the case
body — so Rule 2 doesn't trigger and the bug stays silent there too.

### Net

- **JS compiler:** correct, masks the underlying issue.
- **Rust compiler:** broken for switch case bodies containing no-else
  `if`. Users who run `lykn compile` (Rust binary) on the
  test-equivalent code hit the bug.
- **Test suite:** doesn't catch the divergence because it only runs
  the JS compiler at the lykn-test layer.

## Deliverable 1 — Fix the `switch` profile (MUST)

**File:** `crates/lykn-lang/src/emitter/forms.rs`.

### 1a. Add a new `PositionalThenStatement` variant (MUST)

Extend the `KernelChildProfile` enum at line 2534-2543:

```rust
enum KernelChildProfile {
    /// All non-head children are in Value context.
    AllValue,
    /// All non-head children inherit the parent's context.
    AllParent,
    /// Per-position contexts; children beyond the array length default to Value.
    Positional(&'static [ExprContext]),
    /// Per-position contexts for fixed positions; children beyond the array
    /// length default to Statement. Use this for forms with a fixed prefix
    /// followed by a variable-length sequence of statement-position children
    /// (e.g., `switch`: discriminant=Value, then case sub-lists=Statement).
    PositionalThenStatement(&'static [ExprContext]),
}
```

### 1b. Update `context_for` to handle the new variant (MUST)

Update the impl at line 2546-2555:

```rust
impl KernelChildProfile {
    fn context_for(&self, child_index: usize, parent_ctx: ExprContext) -> ExprContext {
        match self {
            KernelChildProfile::AllValue => ExprContext::Value,
            KernelChildProfile::AllParent => parent_ctx,
            KernelChildProfile::Positional(positions) => positions
                .get(child_index)
                .copied()
                .unwrap_or(ExprContext::Value),
            KernelChildProfile::PositionalThenStatement(positions) => positions
                .get(child_index)
                .copied()
                .unwrap_or(ExprContext::Statement),
        }
    }
}
```

### 1c. Use the new variant for `switch` (MUST)

Update the `switch` entry at line 2584:

```rust
// switch: discriminant=Value, then case sub-lists=Statement
"switch" => KernelChildProfile::PositionalThenStatement(&[V]),
```

(Comment text MUST be preserved or improved — it documents the intent.)

### 1d. Verify no other entries should be `PositionalThenStatement` (MUST)

Audit every other `Positional` entry in `kernel_child_profile`. For each
entry, verify the child structure is fixed-length (no variable-length
trailing sequence). If any entry has variable-length trailing children
that should be in Statement context, convert it to
`PositionalThenStatement`.

The current `Positional` entries (lines 2574-2589) are:

| Form | Profile | Variable-length tail? |
|---|---|---|
| `while` | `Positional(&[V, S])` | No (cond + body, fixed 2) |
| `do-while` | `Positional(&[S, V])` | No (body + cond, fixed 2) |
| `for` | `Positional(&[V, V, V, S])` | No (init + test + update + body, fixed 4) |
| `for-of`/`for-in`/`for-await-of` | `Positional(&[S, V, S])` | No (binding + iterable + body, fixed 3) |
| `switch` | `Positional(&[V])` | **Yes** — variable case count → fix to `PositionalThenStatement(&[V])` |
| `if` | `AllValue` (entry exists for completeness; specially handled at line 375) | No |
| `const`/`let`/`var` | `Positional(&[S, V])` | No (name + init, fixed 2) |

After the audit, **MUST report** in the commit message and closing
report which entries were verified and which (if any) were changed.

### 1e. Verify by reading the regression vector (MUST)

After the fix, trace by hand the test pattern:

```lykn
(switch x (1 (if (= y 0) (console:log "a")) (break)) (default (break)))
```

- `switch` profile: `PositionalThenStatement(&[V])`
- Child 0 (discriminant `x`): index 0 → `positions[0] = V` ✓
- Child 1 (case `(1 ...)`): index 1 → overflow to `Statement` ✓
- Recursing into case `(1 ...)` with parent_ctx = Statement
- Case head `1` is non-atom → line 397-403 fallthrough, no context change
- Case body `(if (= y 0) (console:log "a"))` processed in **Statement** context
- `if` head: line 375-379 interception checks `matches!(ctx.expr_context, Value | Tail)` — Statement does not match, **interception does NOT fire** ✓
- Falls to the kernel-form fallthrough at line 380-396
- `kernel_child_profile("if")` = `AllValue` — children all get Value
- Each child of `if` is processed: `(= y 0)` → expression, `(console:log "a")` → expression
- The list `(if (= y 0) (console:log "a"))` is rebuilt and emitted
- Codegen at `codegen/emit.rs` renders it as JS `if (y === 0) console.log("a");`

After the fix, the test passes for the **right** reason — Rust now
correctly preserves Statement context for switch case bodies, matching
the JS compiler's behavior.

## Deliverable 3 — Cross-compiler test helper + use it for switch tests (MUST)

The lykn-side tests in `dd-50_test.lykn` currently only run the JS
compiler (via `compile` from `testing/helpers.js`). This masks
divergence between the two compilers. The DD-50 implementation plan's
"cross-compiler byte-identical convergence" gate (per
`docs/dev/0009-implementation-plan-dd-50-position-aware-compilation.md`)
is not actually enforced by the test suite.

### 3a. Add a `compile-both` helper (MUST)

**File:** `packages/testing/helpers.js`. Add a new exported function:

```js
/**
 * Compile lykn surface source via BOTH compilers (JS and Rust) and
 * return the JS output. Throws if either compiler produces output that
 * differs in the head + first 200 characters of the result, or if
 * either compiler errors. Use this for cross-compiler convergence tests.
 *
 * The Rust compiler is invoked via the `lykn` binary on the user's
 * PATH (set by the project's build/test setup). If the binary is not
 * available, this function throws a clear error directing the caller
 * to build the Rust crate.
 *
 * @param {string} source - lykn source text
 * @returns {string} compiled JavaScript (from JS compiler — Rust output
 *   verified to match)
 */
export function compileBoth(source) {
  // 1. Compile via JS
  const jsOut = lykn(source).trim();
  // 2. Compile via Rust binary (using a temp file or stdin)
  const rustOut = invokeRustCompiler(source).trim();
  // 3. Compare — if they differ in the substantive output (modulo
  //    whitespace / trailing newlines), throw with a diff.
  if (!convergent(jsOut, rustOut)) {
    throw new Error(
      `cross-compiler divergence:\n--- JS ---\n${jsOut}\n--- Rust ---\n${rustOut}`
    );
  }
  return jsOut;
}
```

The exact implementation of `invokeRustCompiler` and `convergent` is
CC's design choice. Suggested approach: use Deno's
`Deno.Command("lykn", { args: ["compile", "-"], stdin: "piped" })`
to pipe the source to the Rust binary.

If the Rust binary is unavailable in the test environment (not built),
the helper SHOULD throw a clear error message rather than silently
falling back to JS-only — silent fallback is what got us into this
situation.

### 3b. Use `compile-both` for the switch regression test (MUST)

Update the existing switch test (line 129) to use `compileBoth` (or
