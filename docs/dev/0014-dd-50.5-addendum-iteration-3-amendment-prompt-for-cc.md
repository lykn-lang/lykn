### Why the rewrite is valid

At top level, free variables (`x`, `y`, `ready`, etc.) compile to JS
references to globals. The compiled JS contains `undefined`
references at runtime, but at compile time (which is what
`compile-both` tests) the output is valid JS. The `includes` checks
substring presence; runtime semantics are not exercised.

The `block` test similarly: `(block (and x y) (console:log "done"))`
at top level compiles to `{ x && y; console.log("done"); }` — valid
JS expression-statements inside a block-statement.

**File:** `workbench/2026-05-10-DD-50.5-addendum-closing-report.md`,

Add this exact entry (copy verbatim, do not paraphrase or shorten):

```markdown
### Implicit-return-of-statement bug (surfaced 2026-05-10 during iter-3)

`func` without `:returns` blindly wraps the last body expression in
`(return ...)` via `wrapReturnLast`. When the last expression is a
statement-only form (no-else `if`, `while`, `for`, `block`, `try`,
`switch`, `do-while`, etc.), the wrapping produces:

- **JS compiler:** throws Rule 2 ("if in expression position
  requires an else branch") — the wrapper-induced position is
  treated as user-intent expression context, which is incorrect:
  the wrapping is mechanical, not user-authored.
- **Rust compiler:** emits invalid JS `return if (...) ...` (and
  equivalents for other statement-only forms).

Both compilers share the bug. The fix is to make `wrapReturnLast`
skip wrapping when the last expression compiles to a statement-only
form — emit it as a plain statement and either return `undefined`
implicitly or do not return at all (whichever matches the user's
explicit-or-absent `:returns` declaration).

**Scope:** out of DD-50.5's surface area; affects every `func`
without `:returns` whose body ends in a no-else `if`, `while`,
`for`, `block`, `try`, `switch`, or `do-while`. **Severity:
substantive.**

**Discovery context:** surfaced when iter-3's first test draft
prescribed `(func test :args ... :body (if (= x 1) ...))` as the
wrapping for surface-form-inside-if regression tests. The wrapping
triggered the bug; the wrapping was incidental to what iter-3 was
testing. iter-3 amendment removed the wrapping and the tests now
pass. The implicit-return bug is logged here as a future-iteration
finding (DD-50.6 candidate, or fold into M8 as a sibling fix to
the kernel-form context refinement work).
```

This is intentionally detailed because future-you (or future-CC)
will need full context to address the bug without re-discovering it.
