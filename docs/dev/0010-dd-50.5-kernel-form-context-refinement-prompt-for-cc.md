# DD-50.5 — Kernel-Form Context Refinement (Prompt for CC)

**Status:** New task. Not an iteration of DD-50; a separate refinement
that addresses an architectural over-broadness in DD-50's iteration-2
fix. Conceptually isolated from DD-50 (which is closed) — start fresh.

**Context:** DD-50 iteration 2 (commits `8701ada` + `f9a599f`) closed
the V-04/V-05 fix and made the position-aware framework correct for
all adjacent-bug patterns. CDC review accepted iteration 2 on
2026-05-09 with one architectural finding logged for this follow-up
task.

**The finding:** the Rust kernel-form fallthrough in
`crates/lykn-lang/src/emitter/forms.rs` lines 384-391 sets
`ExprContext::Value` for **all** non-head children of every kernel
form. This is correct for expression-producing kernel forms (`?`,
`+`, `throw`, `return`, etc.) but **wrong** for statement-bodied
kernel forms (`while`, `for`, `block`, `try`, `switch`, etc.) whose
body children should remain in `Statement` context.

**The bug:** `(while cond (if c (throw e)))` — a previously-valid
program where the no-else `(if c (throw e))` is the loop body —
triggers Rule 2 compile-error after iter-2 because the loop body is
now in `Value` context. The no-else `if` reaches
`emit_if_expression`, sees no else, emits the COMPILE_ERROR throw.
Same pattern fires for `if` in `for`/`block`/`try`/`switch`/`label`
body positions.

**Scope: focused.** Five MUST deliverables. **All are MUST — no
optional items.** If any deliverable cannot be completed as
specified, stop and surface the constraint to Duncan/CDC.

This task is conceptually clean: introduce a per-kernel-form context
profile, route the fallthrough through it, regression-test the
previously-broken patterns. Modest scope; precise boundaries.

## Deliverable 1 — Audit kernel forms and produce the context profile (MUST)

**Required output:** a constant in
`crates/lykn-lang/src/emitter/forms.rs` named `KERNEL_FORM_CHILD_CONTEXT`
(or a function with the same logic — CC's call) that takes a kernel
form's head atom and the parent's `ExprContext`, and returns the
intended child contexts.

**Required content:** the constant MUST classify every entry in
`crates/lykn-lang/src/classifier/dispatch.rs`'s `is_kernel_form`
match. Three categories:

### Category A — All-Value (children are uniformly Value context)

These are expression-producing kernel forms whose children are all
expression-position. The head dispatches differently but children
are uniform.

MUST include all of:

- All operators: `+`, `-`, `*`, `/`, `%`, `**`, `===`, `!==`, `==`,
  `!=`, `<`, `>`, `<=`, `>=`, `&&`, `||`, `??`, `&`, `|`, `^`, `<<`,
  `>>`, `>>>`, `!`, `~`, `++`, `--`
- Compound assignment: `+=`, `-=`, `*=`, `/=`, `%=`, `**=`, `<<=`,
  `>>=`, `>>>=`, `&=`, `|=`, `^=`, `&&=`, `||=`, `??=`
- Expression operators: `new`, `delete`, `typeof`, `instanceof`,
  `in`, `void`, `await`, `yield`, `yield*`
- Member access: `get`
- Assignment: `=`
- Ternary: `?`
- Literals: `array`, `object`, `template`, `tag`, `regex`, `spread`,
  `rest`, `default`, `alias`
- Quoting: `quote`, `quasiquote`
- Async/dynamic: `async`, `dynamic-import`
- Sequence/comma: `seq`

For these, all non-head children → `Value`.

### Category B — Statement-position bodies (mixed by position)

These have specific positional semantics. The function MUST encode
these per-form:

| Form | Child position semantics |
|---|---|
| `if` | test=Value, then=parent-context, else=parent-context (already specially handled at lines 375-379; the fallthrough should NOT see this case, but include the entry for completeness) |
| `while` | test=Value (boolean), body=Statement |
| `do-while` | body=Statement, test=Value (boolean) |
| `for` | init=Value (or whatever the existing for handler expects), test=Value, update=Value, body=Statement |
| `for-of` | binding=binding-context (no recursion needed for atoms), iterable=Value, body=Statement |
| `for-in` | same as `for-of` |
| `for-await-of` | same as `for-of` |
| `block` | all children=Statement |
| `try` | body=Statement, catch-clause body=Statement, finally body=Statement |
| `switch` | discriminant=Value, case bodies=Statement |
| `label` | label=binding-context, body=Statement |
| `throw` | arg=Value (the throw is a statement; its arg is an expression) |
| `return` | arg=Value if present |
| `break`, `continue` | optional label=binding-context, no expression args |
| `debugger` | no children |
| `import`, `export` | special — preserve current behavior |

For these, the function returns a list-per-position context.

### Category C — Declarations / definitions

These have specific child-context semantics tied to declaration form:

| Form | Child position semantics |
|---|---|
| `const`, `let`, `var` | name=binding, initializer=Value |
| `function`, `function*`, `=>` | name=binding, params=binding, body=context-aware (last expression Value/Tail per existing logic; non-last Statement) |

### Default for unknown kernel forms

If a head atom isn't in the profile (shouldn't happen given the
audit, but for defensive default): use `Value` for all non-head
children. **However**, log a warning (or panic in debug builds) so
it's auditable — unknown kernel-form heads should be added to the
profile, not silently defaulted.

**MUST verify** by running:

```sh
grep -E "matches!" crates/lykn-lang/src/classifier/dispatch.rs
```

and walking every kernel-form entry, confirming it's classified by
the new profile.

## Deliverable 2 — Refactor the kernel-form fallthrough to use the profile (MUST)

**File:** `crates/lykn-lang/src/emitter/forms.rs` lines 380-396.

**Current code (over-broad):**

```rust
} else {
    // Not a surface form — recursively process all subexpressions.
    let saved_ctx = ctx.expr_context;
    let mut new_values = Vec::with_capacity(values.len());
    new_values.push(values[0].clone()); // head atom unchanged
    ctx.expr_context = ExprContext::Value;
    for v in &values[1..] {
        new_values.push(emit_expr(v, ctx, registry));
    }
    ctx.expr_context = saved_ctx;
    SExpr::List { values: new_values, span: *span }
}
```

**Required replacement:** route through the profile from
Deliverable 1.

```rust
} else {
    let saved_ctx = ctx.expr_context;
    let child_contexts = kernel_form_child_contexts(head_name, saved_ctx);
    let mut new_values = Vec::with_capacity(values.len());
    new_values.push(values[0].clone()); // head atom unchanged
    for (i, v) in values[1..].iter().enumerate() {
        ctx.expr_context = child_contexts.get(i).copied().unwrap_or(ExprContext::Value);
        new_values.push(emit_expr(v, ctx, registry));
    }
    ctx.expr_context = saved_ctx;
    SExpr::List { values: new_values, span: *span }
}
```

(The exact API of `kernel_form_child_contexts` is CC's design choice
— inline match, vec-returning function, or const lookup. The
constraint: it MUST return per-child contexts, not a single
"propagate this to all children" decision.)

**MUST verify** that the refactor preserves DD-50's existing
behavior for V-04/V-05 cases. The 16 tests in `dd-50_test.lykn`
MUST all still pass without modification.

All 7 patterns MUST pass. The negative patterns (1-5) confirm

## Deliverable 4 — JS-side parity (MUST)

**File:** `packages/lang/compiler.js`.

The JS compiler doesn't have an explicit "kernel-form fallthrough"
the way Rust does — instead, each macro handler decides what to pass
to `compileExpr`. The DD-50 iter-2 site audit updated 64 call sites
to pass `'expression'`. CC MUST audit those changes against the
same statement-bodied-form principle:

**Required action:** find every macro handler in `compiler.js` for a
statement-bodied kernel form (`while`, `for`, `for-of`, `for-in`,
`for-await-of`, `do-while`, `block`, `try`, `switch`, `label`) and
verify their body-position child compilations use the default
(`'statement'`) — NOT `'expression'`.

The check pattern: for each statement-bodied form's macro handler,
the body children should appear as `compileExpr(e)` (no second arg)
or `toStatement(compileExpr(e))` — NOT `compileExpr(e, 'expression')`.

If iter-2's audit incorrectly passed `'expression'` to
statement-position children, fix those sites.

**Required regression tests** in `test/forms/dd-50.test.js`,
mirroring 5 of the 7 lykn-side patterns above:

```js
Deno.test("DD-50.5: no-else if in while body preserved as statement", () => {
  const out = lykn('(while true (if false (throw (new Error "e"))))');
  // Should NOT throw at compile time. Should produce valid `if` statement.
  assertStringIncludes(out, "while (true)");
  assertStringIncludes(out, "if (false)");
  assertStringIncludes(out, "throw");
});

Deno.test("DD-50.5: no-else if in for body preserved as statement", () => {
  const out = lykn('(for (let i 0) (< i 10) (++ i) (if (= i 5) (break)))');
  assertStringIncludes(out, "for (");
  assertStringIncludes(out, "break");
});

Deno.test("DD-50.5: no-else if in block preserved as statement", () => {
  const out = lykn('(func test :args (:any x) :body (if (= x 1) (console:log "yes")) (console:log "done"))');
  assertStringIncludes(out, "if (x === 1)");
});

Deno.test("DD-50.5: no-else if in try body preserved as statement", () => {
  const out = lykn('(try (if (= x 1) (throw (new Error "e"))) (catch e (console:log e)))');
  assertStringIncludes(out, "try");
  assertStringIncludes(out, "if (x === 1)");
});

Deno.test("DD-50.5: positive — if in return arg still intercepted", () => {
  const out = lykn('(func test :returns :any :body (return (if cond "a" "b")))');
  assertStringIncludes(out, 'return cond ? "a" : "b"');
});
```

## Anticipated objections

- **"DD-50's tests pass; why refine?"** Because the silent
  regression vector affects user code patterns that aren't in the
  test suite — `(while cond (if c (throw e)))` and similar. The
  bug is silent: the user gets a confusing compile-error throw at
  runtime instead of valid loop+if-statement output. The fix is
  small and bounded.
- **"The fallthrough is one place; per-form profile adds
  complexity."** True — the profile adds a small classification
  table (≤30 entries). The complexity is bounded and scopes
  correctly with the kernel-form vocabulary. The alternative
  (silent-regression for statement-bodied forms) is worse for
  users.
- **"Why not address all the M7-3 closing-report fast-follows in
  this task?"** Out of scope by design. This task addresses ONLY
  the kernel-form-fallthrough over-broadness. Other fast-follows
  (analysis-pass replacement for Rule 2, surface-forms guide
  documentation, SKILL.md style guidance) are separate tasks.
