# DD-50.5 Addendum — Iteration 2 Prompt for CC

## Read this section carefully before doing any work

The DD-50.5 addendum (commit `3db3e1f`) successfully fixed the `switch`
profile and added the `compileBoth()` cross-compiler test helper. CDC
review on 2026-05-10 verified those parts. **But the closing report
contains a misdiagnosis that needs to be corrected before DD-50.5
closes.**

### What CC reported

> "Surface = vs === (pre-existing) — Rust emits = (assignment) where
> JS emits === (equality) for `(= a b)` in certain kernel contexts.
> Logged as fast-follow for M8."

To work around this "divergence," CC modified two existing tests in
`test/forms/dd-50_test.lykn` to use `===` directly instead of surface
`=`:

- For test (line 109): `(if (=== i 5) (break))` — was `(if (= i 5) (break))`
- Switch test (line 127): `(if (=== y 0) ...)` — was `(if (= y 0) ...)`

### What's actually true about `=` in lykn

CDC verification on 2026-05-10 (Explore agent trace, plus reading the
relevant docs and code) established:

**Surface `(= a b)` IS equality. It compiles to `a === b` on BOTH
compilers.** The design is documented:

- **DD-22** in `docs/design/` commits to surface `=` as equality.
- **`docs/guides/00-lykn-surface-forms.md`** documents surface `(= a b)`
  as the equality form.
- **`assets/ai/SKILL.md`** "Anti-patterns" table reinforces: "Using
  `===`/`==` kernel operators directly | Use `(= a b)` for equality —
  it compiles to `===`. Use `(js:eq x null)` for the `== null` idiom."

The implementation matches the design:

- **JS:** `packages/lang/surface.js` (around line 1193-1210) — surface
  `=` macro transforms `(= a b)` → kernel `(=== a b)`.
- **Rust:** `crates/lykn-lang/src/classifier/dispatch.rs:30` lists `=`
  in `is_surface_form`. `classifier/forms.rs::classify_eq()` produces
  `SurfaceForm::Eq`. `emitter/forms.rs::emit_eq()` (around line 705)
  emits kernel `(=== a b)`.

Both compilers correctly route surface `=` to equality. **There is
no divergence in `=` semantics.** The "fast-follow for M8" logged in
the closing report does not refer to a real bug.

### What this means

1. **CC's diagnosis was wrong.** The fix in tests (switching to `===`)
   was based on a misreading of what was actually broken.
2. **The actual divergence is still unknown.** `compileBoth()` DID
   throw on the original tests with `(= i 5)` and `(= y 0)` — that's
   why CC made the changes. But the CAUSE of those throws was NOT the
   `=` semantics. Something else differs between JS and Rust output
   for those test patterns. We don't know what.
3. **The test workarounds mask the unknown bug.** Whatever made
   `compileBoth()` throw is still in the codebase. Reverting the
   workarounds will let `compileBoth()` re-throw with the actual
   divergence so we can diagnose it properly.
4. **The "fast-follow for M8" is a phantom item.** It needs to be
   retracted from the closing report and replaced with whatever the
   real finding is once we know it.

### Why CC may have gotten this wrong

This is recorded in `assets/ai/SKILL.md` and surface-forms-guide as
load-bearing language design. The likely failure mode: CC saw the
divergence error from `compileBoth()`, read the JS output as `===`
and the Rust output as something containing `=`, and pattern-matched
to "kernel `=` is assignment, surface `=` is equality" — concluding
that one compiler must be applying the surface transform and the
other not. But the actual JS and Rust output for a working `(= i 5)`
in equality context should both be `===`. The divergence pattern was
something else entirely.

**The lesson worth internalizing:** when a cross-compiler check fires,
do NOT assume you know what the divergence is from a quick read. Run
the compilers explicitly, capture both outputs, do a literal byte-by-
byte diff. Then diagnose. CC took a shortcut and the shortcut was
wrong.

## Deliverable 3 — Diagnose the root cause (MUST)

Based on the captured outputs from Deliverable 2, identify the actual
source of any divergence. Possibilities to consider, in rough order
of likelihood:

1. **Whitespace / formatting beyond what `normalize()` catches.** The
   helper's normalize collapses whitespace and `; }` patterns but
   may miss other patterns (e.g., line breaks inside expressions,
   trailing newlines, parenthesization differences).
2. **Brace handling.** JS may emit `for (...) { if (...) break; }`
   while Rust emits `for (...) if (...) break;` (no braces). That's
   a structural difference, not a semantic one — both are valid JS.
3. **For-loop init form.** `(let i 0)` may emit `let i = 0` in one
   compiler and `var i = 0` in the other (or different).
4. **Update expression form.** `(++ i)` may emit `i++` in one and
   `++i` in the other.
5. **Switch case structure.** Rust and JS may produce slightly
   different switch-case shapes (e.g., default case position,
   case-body block-wrapping).
6. **A genuine bug in one of the compilers** unrelated to the above
   — surface a specific kernel form being emitted incorrectly.

**MUST identify which (if any) is the actual cause.** Document your
investigation steps. Do NOT assume you know — read the captured
outputs literally.

For each divergence found, classify it as:

- **(A) Cosmetic** — both outputs are valid JS with equivalent
  semantics; the divergence is in formatting or non-semantic
  structure. Fix: tighten `compileBoth()`'s `normalize()` to
  recognize the equivalence, OR accept the divergence with a
  documented exception list.
- **(B) Semantic but harmless** — outputs differ structurally but
  produce the same runtime behaviour for the test pattern. Same
  treatment as cosmetic, but more carefully documented.
- **(C) Genuine bug in one compiler** — one output is wrong, one
  is right per the spec. Fix: identify which compiler is wrong,
  fix the bug, log as a substantive finding.
