# DD-50.5 Addendum — Iteration 3 Prompt for CC

## Context

Iteration 2 (commit `804425e`) correctly diagnosed and fixed two real
bugs:

1. **`KernelPassthrough` skipped emitter descent.** Surface forms (`=`,
   `and`, `or`, `not`) nested inside kernel forms (`if`, `while`,
   `for`, `block`, `try`, `switch`) were never transformed. Fixed in
   `crates/lykn-lang/src/emitter/forms.rs` by changing
   `SurfaceForm::KernelPassthrough { raw, .. } => vec![raw.clone()]`
   to `vec![emit_expr(raw, ctx, registry)]`.
2. **`compileBoth()` used stale PATH binary.** Fixed to use
   `LYKN_BIN` env var with `./bin/lykn` fallback (which `make build`
   produces).

The fix is correct. **What's missing is regression coverage.**

The current test suite only exercises the fix for one
surface-in-kernel pattern: `(if (= i 5) (break))`. The
`KernelPassthrough` recursive-descent fix mechanically affects every
surface form nested inside every kernel control-flow form — a much
broader class than `=` inside `if`. Without targeted tests, a future
refactor of `emit_form` (or a regression in `KernelPassthrough`
handling) could re-break some subset of the matrix and the test suite
wouldn't catch it.

This iteration adds the regression coverage. Small, focused, bounded.

## Anticipated objections

- **"Iteration 2's tests already pass; why add more?"** Iteration 2's
  test exercises ONE cell of the matrix (`=` in `if` test). The
  `KernelPassthrough` fix's mechanism extends to the whole matrix.
  Without the broader tests, a future refactor of `emit_form` or
  `KernelPassthrough` could silently re-break some cells. The matrix
  is the regression gate.
- **"Full cartesian would be 30 tests; 11 is incomplete."** The
  cartesian is 30; the row+column cover is 11. Since the fix's
  mechanism is uniform (all forms go through the same `emit_expr`
  recursion), testing every row and every column at least once
  catches regressions in the mechanism. Testing the full cartesian
  would catch combinatorial bugs, but the fix doesn't have a
  combinatorial structure — it's uniform. 11 is the right size.
- **"Why not other surface forms like `do`, `swap!`, etc.?"** The
  five tested (`=`, `!=`, `and`, `or`, `not`) are the most common
  surface forms appearing inside kernel-form condition/body
  positions — these are where the bug manifested. `do` is itself a
  control-flow form (handled by DD-50.5 Rule 4). `swap!`/`reset!`
  are statement-position mutation ops and appear in different
  contexts. Adding tests for those would be `iter-4` if needed,
  but they're a different surface area. Keep iter-3 tight.
