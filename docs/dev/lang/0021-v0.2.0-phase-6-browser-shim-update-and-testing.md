## Implementation Order

### 1. Fix `node:path` browser incompatibility

Option A (recommended): Dynamic import with fallback

```js
let nodePath;
try { nodePath = await import("node:path"); } catch { nodePath = null; }
```

But expander is sync. Better: use esbuild `--external:node:path` and provide a browser shim that makes `pass0ImportMacros` throw on `import-macros` in browser.

Simplest approach: in `build.js`, add `external: ["node:path"]` and in `pass0ImportMacros`, catch the case where `resolve`/`dirname` are unavailable.

Actually simplest: extract the `node:path` usage into a try/catch at module level:

```js
let resolvePath, dirnamePath;
try {
  const path = await import("node:path"); // top-level await
  resolvePath = path.resolve;
  dirnamePath = path.dirname;
} catch {
  // Browser environment — import-macros not available
}
```

Then in `pass0ImportMacros`, if `resolvePath` is null, throw: "import-macros not available in browser".

BUT: `src/expander.js` uses `import { resolve, dirname } from "node:path"` as a static import. esbuild bundles it. The fix for esbuild: add `external: ["node:path"]` to the build config, and provide inline fallbacks.
