# DD-09: v0.1.0 Scope and Phasing

## Your role

You are helping design lykn, an s-expression syntax for JavaScript.
Read the session bootstrap doc in this project for full context. This
conversation synthesizes all prior design decisions (DD-01 through
DD-08) into a concrete implementation plan for v0.1.0.

**Important**: Read ALL prior decision docs (DD-01 through DD-08)
before starting this discussion. This topic cannot be done until the
others are settled.

## Context

Lykn v0.0.1 is a minimal PoC that handles 22 ESTree node types.
The gap analysis identified ~31 missing node types, grouped into
Essential (15 capabilities), Important (16), and Nice-to-have (11).

The prior 8 discussions decided the syntax for all major forms. Now
we need to:

1. Define what's IN v0.1.0 and what's NOT
2. Order the implementation work
3. Identify dependencies between forms
4. Set expectations for what v0.1.0 enables

## What v0.0.1 already handles

Forms: `var`, `const`, `let`, `.`, `=>`, `lambda`, `return`, `if`,
`block`, `=`, `new`, `array`, `object`, all binary/logical operators,
all unary prefix operators, function calls.

ESTree nodes: `ArrayExpression`, `ArrowFunctionExpression`,
`AssignmentExpression`, `BinaryExpression`, `BlockStatement`,
`CallExpression`, `EmptyStatement`, `ExpressionStatement`,
`FunctionExpression`, `Identifier`, `IfStatement`, `Literal`,
`LogicalExpression`, `MemberExpression`, `NewExpression`,
`ObjectExpression`, `Program`, `Property`, `ReturnStatement`,
`UnaryExpression`, `VariableDeclaration`, `VariableDeclarator`.

## Questions to discuss

1. **What's the bar for v0.1.0?** Should it be "can write a real
   program" (e.g., a small Express server, a CLI tool)? Or "feature
   complete for modern JS"? Or somewhere in between?

2. **Proposed v0.1.0 scope** (adjust based on prior decisions):

   Definitely in:
   - Colon syntax for member access (DD-01)
   - camelCase conversion (DD-01)
   - `this` / `super` as proper nodes (DD-08)
   - Function declarations (DD-02)
   - `throw`, `try`/`catch`/`finally` (from gap analysis)
   - `while`, `for`, `break`, `continue` (from gap analysis)
   - `import` / `export` (DD-04)
   - Compound assignment operators (DD-08)
   - `async` / `await` (DD-03)
   - Template literals (DD-05)
   - Destructuring patterns (DD-06)
   - Spread / rest
   - `for-of`, `for-in`
   - `?:` ternary (DD-08)
   - `switch`/`case`

   Maybe in:
   - Classes (DD-07)
   - Update operators `++`/`--` (DD-08)
   - Default parameters
   - `do-while`
   - `seq`
   - `debugger`
   - `label`
   - `regex`
   - Dynamic `import()`
   - `**` exponentiation

   Definitely out (v0.2.0+):
   - Generators / `yield`
   - `for await...of`
   - Optional chaining `?.`
   - Private fields
   - Static blocks
   - Class fields (PropertyDefinition)
   - MetaProperty
   - BigInt
   - Logical assignment operators

3. **Implementation order**: What should be built first? Dependencies:
   - Colon syntax and camelCase are foundational → first
   - `this`/`super` depend on colon syntax → after DD-01
   - Function declarations are independent → early
   - `import`/`export` are independent → early
   - `async`/`await` depends on function forms → after DD-02
   - Destructuring depends on function params → after DD-02
   - Classes depend on everything → late
   - Template literals are independent → any time

4. **Testing strategy**: Should each form have its own test file?
   Integration tests that compile a small program? Both?

5. **What does "v0.1.0 is done" look like?** A specific program that
   compiles and runs? A set of examples? A test suite passing?

6. **Reader changes**: Do any decisions from DD-01 through DD-08
   require changes to the reader, or is everything compiler-only?

7. **Browser shim**: Is the browser shim (`<script type="text/lykn">`)
   part of v0.1.0, or a separate track?

## Goal

By the end of this discussion, produce:

- An ordered implementation plan (what to build in what sequence)
- A "done" definition (specific programs or tests that must work)
- Timeline expectations (rough sizing, not calendar dates)
