# DD-25.1: Nested Destructuring and Defaults in `func`/`fn` Parameters

**Status**: Decided (implementation deferred until DD-25 lands)

Two features deferred from DD-25, now designed and ready to
implement as a follow-up. DD-25 established the `ParamShape` type,
the accessor pattern, and the parsing infrastructure. DD-25.1
extends the parser to accept nested patterns and `default` sub-forms
that DD-25 explicitly recognizes-and-rejects with helpful errors.

**Combined LoE**: ~5–7 hours (both features, both compilers, tests).
Single PR after DD-25 is merged and green.

## Feature 1: Nested destructuring in parameters

### Syntax

```lisp
;; Nested object in object
(func process-order
  :args ((object :string id
                 (alias :any customer (object :string name :string email))
                 :number total)
         :string action)
  :returns :string
  :body (template name " (" email ") — $" total " — " action))

;; Nested object in array
(func process-pair
  :args ((array (object :string name) (object :string name)))
  :body (console:log name name))

;; Multiple nesting levels
(func deep
  :args ((object (alias :any address
                   (object :string city
                           (alias :any geo (object :number lat :number lng))))))
  :body (console:log city lat lng))
```

### Compiled output

```javascript
// Dev mode
function processOrder({id, customer: {name, email}, total}, action) {
  if (typeof id !== "string")
    throw new TypeError("processOrder: arg 'id' expected string, got " + typeof id);
  if (typeof name !== "string")
    throw new TypeError("processOrder: arg 'name' expected string, got " + typeof name);
  if (typeof email !== "string")
    throw new TypeError("processOrder: arg 'email' expected string, got " + typeof email);
  if (typeof total !== "number" || Number.isNaN(total))
    throw new TypeError("processOrder: arg 'total' expected number, got " + typeof total);
  if (typeof action !== "string")
    throw new TypeError("processOrder: arg 'action' expected string, got " + typeof action);
  return `${name} (${email}) — $${total} — ${action}`;
}

// --strip-assertions
function processOrder({id, customer: {name, email}, total}, action) {
  return `${name} (${email}) — $${total} — ${action}`;
}
```

### How nesting enters the parser

DD-25's `parseDestructuredParam` currently rejects lists in the
name position with "nested destructuring not yet supported." The
change: instead of rejecting, **recursively call
`parseDestructuredParam`** when a list starting with `object` or
`array` appears in name position.

The recursive call is identical to the top-level call — it parses
`:type name` pairs inside the nested pattern. The recursion
terminates because each level consumes its closing paren.

### Where nesting appears syntactically

Nesting uses `alias` to name the intermediate binding (so the
kernel can emit the rename):

```lisp
(object :string name
        (alias :any addr (object :string city :string zip)))
;;       ^^^alias sub-form: addr is the JS property name,
;;       (object ...) is the nested pattern
```

Without `alias`, the nested pattern has no property name to
destructure from:

```lisp
;; ERROR: nested pattern without alias — which property does this destructure?
(object :string name (object :string city))
```

**Rule**: Nested patterns MUST be wrapped in `alias`. The outer
name is the property key; the inner pattern is the destructuring
target. This is consistent with DD-06 kernel destructuring where
`(alias data (object x y))` → `data: { x, y }`.

Exception: array destructuring doesn't need `alias` because
elements are positional:

```lisp
;; OK: array elements are positional, no alias needed
(array :number first (object :string name :string email) :number last)
```

### AST changes

**None.** `ParamShape::DestructuredObject.fields` is
`Vec<TypedParam>`. For nesting, the "name" of a `TypedParam`
becomes... not a string but a nested `ParamShape`. This requires
a small change:

```rust
// In ParamShape's field representation, allow nesting:
#[derive(Debug, Clone, PartialEq)]
pub enum DestructuredField {
    /// :type name — flat field
    Simple(TypedParam),
    /// (alias :type name (object/array ...)) — nested pattern
    Nested {
        alias_name: String,
        type_keyword: String,  // always :any for the intermediate
        pattern: Box<ParamShape>,
        span: Span,
    },
}
```

Update `DestructuredObject.fields` from `Vec<TypedParam>` to
`Vec<DestructuredField>`.

**Accessor impact**: `typed_params()` recurses into
`DestructuredField::Nested` to collect leaf-level typed params.
`bound_names()` does the same. `to_kernel()` emits nested
`(alias name (object ...))` forms. The accessor methods absorb
the complexity — downstream code doesn't change.

### Type checks for nested patterns

Type checks are emitted for **leaf fields only**. Intermediate
objects (the `alias` targets) get `:any` — they're structural
containers, not typed values. The type checks fire after JS
runtime destructuring has extracted the values, so they see the
final leaf values.

```lisp
(object (alias :any addr (object :string city :number zip)))
;; Type checks emitted for city (string) and zip (number)
;; No type check for addr itself — it's :any
```

### Scope tracking

`bound_names()` returns leaf names only. Intermediate alias
names (`addr`) are also bound (they're `const` bindings in the
compiled JS) and should be tracked. Update `bound_names()` to
include both alias names and leaf names.

### Tests

| Test case | What it verifies |
|-----------|-----------------|
| Nested object in object with alias | Correct kernel form + type checks |
| Nested object in array (no alias) | Positional nesting works |
| Two levels deep | Recursive parsing terminates correctly |
| `:any` at intermediate level | No type check for intermediate |
| `:any` at leaf level | No type check for leaf |
| Nested without alias on object | Compile error with helpful message |
| `--strip-assertions` | Type checks stripped, nesting preserved |

## Feature 2: `default` in destructured fields

### Syntax

```lisp
;; Default value for a field
(func create-user
  :args ((object :string name
                 (default :number age 0)
                 (default :string role "viewer")))
  :returns :object
  :body (obj :name name :age age :role role))

;; In array destructuring
(func pad-triple
  :args ((array :number first
                (default :number second 0)
                (default :number third 0)))
  :body (+ first second third))

;; In fn
(bind f (fn ((object :string name (default :number age 0)))
  (console:log name age)))
```

### Compiled output

```javascript
// Dev mode
function createUser({name, age = 0, role = "viewer"}) {
  if (typeof name !== "string")
    throw new TypeError("createUser: arg 'name' expected string, got " + typeof name);
  if (typeof age !== "number" || Number.isNaN(age))
    throw new TypeError("createUser: arg 'age' expected number, got " + typeof age);
  if (typeof role !== "string")
    throw new TypeError("createUser: arg 'role' expected string, got " + typeof role);
  return { name, age, role };
}

// --strip-assertions
function createUser({name, age = 0, role = "viewer"}) {
  return { name, age, role };
}
```

### How `default` enters the parser

DD-25's parser currently rejects `(default ...)` in destructured
params with "default values in destructured params not yet
supported." The change: recognize `(default :type name value)` as
a 3-element sub-form.

```
Inside a destructured param, the parser sees either:
  1. :type name          → standard field (step 2)
  2. (object ...)        → nested pattern (Feature 1, step 1)
  3. (array ...)         → nested pattern (Feature 1, step 1)
  4. (default :type name value) → defaulted field (this feature, step 1)
  5. (rest :type name)   → rest element (DD-25, step 1)
  6. (alias ...)         → alias sub-form (DD-25 + Feature 1)
  7. _                   → skip (array only, DD-25)
```

The `default` sub-form is consumed as one unit (step 1). Inside
it, the parser expects exactly three elements: type keyword, name,
default value expression.

### AST changes

Add `default_value` to the field representation:

```rust
// Option 1: Add to TypedParam (simplest)
#[derive(Debug, Clone, PartialEq)]
pub struct TypedParam {
    pub type_keyword: String,
    pub name: String,
    pub default_value: Option<SExpr>,  // NEW
    pub span: Span,
}

// This means every TypedParam can optionally have a default,
// which is correct — simple params in :args could theoretically
// have defaults too (currently supported via DD-06's (default)
// form in the kernel).
```

This is the minimal change. `TypedParam` gains an `Option<SExpr>`
field. Existing code that constructs `TypedParam` passes `None`.
The parser populates `Some(value)` when it encounters a `default`
sub-form.

### Accessor impact

`typed_params()` — unchanged. Returns `&TypedParam` which now
*may* have a `default_value`. Downstream code that doesn't care
about defaults (scope tracking, overlap detection) ignores it.

`to_kernel()` — when a field has `default_value: Some(val)`, emit
`(default name val)` instead of bare `name` in the kernel
destructuring pattern. DD-06 already handles `(default name val)`
in kernel destructuring.

`type_checks()` — unchanged. The type check fires on the final
value (after JS applies the default). No special handling needed.

### `default` with `:any`

`(default :any name "fallback")` — no type check, but the default
value is still emitted. The default is runtime semantics (not an
assertion), so it survives `--strip-assertions`.

### `default` interaction with `--strip-assertions`

The default value is **preserved**. It's part of the destructuring
pattern, not an assertion. Only the type checks are stripped.

```javascript
// --strip-assertions: default preserved, type check gone
function createUser({name, age = 0, role = "viewer"}) {
  return { name, age, role };
}
```

### `default` in non-destructured params

This DD covers `default` inside *destructured* params only. For
non-destructured params, `func` already supports `(default :type
name value)` in `:args` via DD-06/DD-16. If it doesn't, that's a
separate gap. Verify and note.

### Tests

| Test case | What it verifies |
|-----------|-----------------|
| Single defaulted field | `{name, age = 0}` pattern correct |
| Multiple defaulted fields | All defaults in one pattern |
| Mixed default + non-default fields | Ordering preserved |
| Default with `:any` | No type check, default preserved |
| `--strip-assertions` | Type checks stripped, defaults preserved |

## Implementation order

Both features in a single PR, after DD-25 is merged:

1. **AST changes first**: `DestructuredField` enum (nesting) +
   `default_value: Option<SExpr>` on `TypedParam` (defaults).
   Both are additive — no existing code breaks.

2. **Parser updates**: Extend `parseDestructuredParam` /
   `parse_destructured_param` to handle both new sub-forms.
   Change the DD-25 error messages from "not yet supported" to
   actual parsing.

3. **Accessor updates**: `typed_params()`, `bound_names()`,
   `to_kernel()` handle the new variants.

4. **Emission**: `to_kernel()` emits nested `alias`/`object`/
   `array` forms and `default` forms. Type checks: leaf fields
   only.

5. **JS compiler**: Parallel changes, same patterns.

6. **Tests**: Full matrix for both features across both compilers.

7. **Cross-compiler**: Verify JSON output matches.

8. **Book**: Update Ch 15 to show nested destructuring and defaults
   working.

## Edge cases

| Case | Behavior |
|------|----------|
| Default on a nested pattern | Not supported: `(default :any addr (object ...))` is ambiguous — is the default the whole nested pattern or the alias? Reject with error. |
| Default value referencing another param | Valid at runtime: `(default :number y x)` → `y = x`. JS evaluates defaults left-to-right. |
| Default value with side effects | Valid but discouraged: `(default :number id (next-id))`. Linter warning in future. |
| Nested pattern without alias in object | Compile error: "nested pattern must use alias to specify the property name" |
| Nested pattern in array (no alias) | Valid: array elements are positional |
| Deeply nested (3+ levels) | Valid: recursion handles arbitrary depth. But linter may warn on depth > 2. |
| Default + rest in same array pattern | Valid: `(array (default :number first 0) (rest :number others))` |
| rest with default | Not supported: `(rest (default :number x 0))` is nonsensical — rest collects all remaining. Compile error. |
