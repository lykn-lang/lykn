# DD-09: v0.1.0 Scope and Phasing

**Status**: Decided
**Date**: 2026-03-24
**Session**: (this chat)

## Summary

v0.1.0 targets "can write a real program" — a small HTTP server, a CLI tool, a utility module. All syntax decisions from DD-01 through DD-08 are included. Implementation is ordered in six phases by dependency: foundation (colon syntax, camelCase), core forms, expressions, destructuring patterns, classes, and browser shim. The reader is unchanged — all work is in the compiler. The browser shim (compile-then-eval via `<script type="text/lykn">`) is included in scope.

## Decisions

### v0.1.0 bar: "can write a real program"

**Decision**: v0.1.0 is the release where lykn stops being a toy. A developer should be able to write a small Express server, a CLI tool, or a utility module in lykn and have it compile to clean, runnable JS. Not "feature complete for modern JS" — that's v1.0 territory.

**Rationale**: The PoC (v0.0.1) proved the architecture. v0.1.0 proves the language is usable. Everything after that is filling gaps.

### What's IN v0.1.0

**Foundation (DD-01)**:
- Colon syntax for static member access (`obj:method`)
- `get` form for computed/dynamic access (`(get obj key)`)
- camelCase conversion with full edge case handling
- `this` → `ThisExpression`
- `super` → `Super`
- Leading `:` reserved (compile-time error)
- `` ` ``, `,`, `#` reserved in reader for future macro system

**Function forms (DD-02)**:
- `function` for named declarations (`FunctionDeclaration`)
- `lambda` for anonymous expressions (`FunctionExpression`)
- `=>` for arrow functions (`ArrowFunctionExpression`)

**Async/await (DD-03)**:
- `async` wrapper on any function form
- `await` unary form
- Top-level await (free — `sourceType: "module"`)

**Modules (DD-04)**:
- Unified `import` form (module path first, then bindings)
- Unified `export` form (wraps declarations, `default`, `names`, re-exports)
- `alias` for renaming in import/export
- `dynamic-import` for expression-level import
- Namespace imports (`import *`) banned
- Re-export-all (`export *`) banned

**Template literals (DD-05)**:
- `template` form with type-based dispatch
- Compiler infers empty `TemplateElement` nodes at boundaries
- `tag` form for tagged templates

**Destructuring (DD-06)**:
- `object` / `array` as patterns (constructor-as-destructor)
- `alias` for rename (reused from DD-04)
- `default` for default values (destructuring and function params)
- `rest` for collecting remaining elements
- `spread` for spreading in expressions
- `_` for skipping array elements
- Nested destructuring via `alias` target
- Destructuring in assignment (`=`), declarations, and function params

**Classes (DD-07)**:
- `class` for declarations, `class-expr` for expressions
- Superclass in list: `(class Dog (Animal) ...)`
- Methods as bare lists (no marker)
- `constructor` as regular method name
- `get` / `set` keyword markers for accessors
- `static` wrapper (composes with `async`)
- `field` for class fields
- Private fields/methods via `-` prefix → `#_` in JS output

**Misc forms (DD-08)**:
- `?` for conditional expression
- Prefix `++` / `--`
- All compound assignment operators (including `&&=`, `||=`, `??=`, `**=`)
- `**` exponentiation
- `debugger`
- `label`
- `do-while` (test first)
- `seq`
- `regex`

**Control flow (from gap analysis)**:
- `throw`
- `try` / `catch` / `finally`
- `while`
- `for`
- `for-of`
- `for-in`
- `break` / `continue`
- `switch` / `case`

**Browser shim**:
- `<script type="text/lykn">` tag scanning
- Compile-then-eval (no interpreter needed)
- Bundled compiler (reader + compiler + astring as single file)
- Error reporting to console

### What's OUT (v0.2.0+)

| Feature | Why deferred |
|---------|-------------|
| Generators / `yield` | Complex, needs iterator protocol design |
| `for await...of` | Depends on generators |
| Optional chaining `?.` | Needs reader/compiler design for `?.` syntax |
| Postfix `++`/`--` | Naming unresolved, rarely needed |
| Named function expressions | Rare use case |
| Named class expressions | Rare use case |
| Computed method names | Interacts with iterator/generator protocol |
| Static blocks | Rarely needed |
| MetaProperty (`new.target`, `import.meta`) | Niche |
| BigInt literals | Niche, needs reader support for `n` suffix |
| Macro system / quasiquote / reader macros | Major feature, needs its own design phase |

### Reader changes: none

**Decision**: The reader is unchanged for v0.1.0. All new functionality is compiler-only.

**Rationale**: Every decision from DD-01 through DD-08 was designed to avoid reader changes. Colon splitting is compile-time. Pattern forms use existing list syntax. Template literals are a compiler form. Private fields use the `-` prefix which the reader already handles as a normal atom character. This also means the Rust reader/formatter needs no changes.

**Implementation note**: Verify during implementation that the reader handles `?` and `#` as valid atom characters. `?` is needed for predicate names (`done?`) and the ternary form. `#` should be accepted by the reader but reserved — the compiler should error on `#`-prefixed atoms with "reserved for future use."

### Implementation order

**Decision**: Six phases, ordered by dependency.

#### Phase 1 — Foundation

No dependencies. Must be done first because everything else builds on it.

| Work item | ESTree nodes | Notes |
|-----------|-------------|-------|
| camelCase conversion | `Identifier` | Character-walk implementation, full edge case table from DD-01 |
| Colon splitting | `MemberExpression` | Split atom on `:`, build chained member expressions |
| `get` form | `MemberExpression` (computed) | Two-arg form, `computed: true` |
| `this` | `ThisExpression` | Special atom dispatch |
| `super` | `Super` | Special atom dispatch |
| Remove `.` form | — | Replace with `get`, update existing tests |

**Estimated effort**: ~1 day

#### Phase 2 — Core forms

Depends on Phase 1 (colon syntax, camelCase). Most items are independent of each other — can be built in any order within the phase.

| Work item | ESTree nodes | Notes |
|-----------|-------------|-------|
| `function` declarations | `FunctionDeclaration` | Name required, params list, multi-statement body |
| `import` unified form | `ImportDeclaration`, `ImportSpecifier`, `ImportDefaultSpecifier` | Module path first, dispatch on arg shape |
| `export` unified form | `ExportNamedDeclaration`, `ExportDefaultDeclaration`, `ExportSpecifier` | Wraps declarations, `default`, `names`, re-exports |
| `alias` in imports/exports | `ImportSpecifier`, `ExportSpecifier` | Rename sub-form |
| `dynamic-import` | `ImportExpression` | Expression-level, separate from `import` |
| `throw` | `ThrowStatement` | Unary form |
| `try`/`catch`/`finally` | `TryStatement`, `CatchClause` | Nested form |
| `while` | `WhileStatement` | Test + body |
| `do-while` | `DoWhileStatement` | Test first (internal consistency) |
| `for` | `ForStatement` | Init, test, update, body |
| `for-of` | `ForOfStatement` | Binding + iterable + body |
| `for-in` | `ForInStatement` | Binding + object + body |
| `break` / `continue` | `BreakStatement`, `ContinueStatement` | Optional label |
| `switch` / `case` | `SwitchStatement`, `SwitchCase` | Test + cases |
| `?` ternary | `ConditionalExpression` | Three args |
| `++` / `--` prefix | `UpdateExpression` | Prefix only |
| `**` exponentiation | `BinaryExpression` | Add to operator table |
| All compound assignments | `AssignmentExpression` | Register each operator |
| `label` | `LabeledStatement` | Name + body, camelCase applies |
| `debugger` | `DebuggerStatement` | No args |
| `seq` | `SequenceExpression` | Two or more args |
| `regex` | `Literal` (regex) | One or two string args |

**Estimated effort**: ~3-4 days

#### Phase 3 — Expressions

Depends on Phase 1. Independent of Phase 2 — can be parallelized.

| Work item | ESTree nodes | Notes |
|-----------|-------------|-------|
| `template` form | `TemplateLiteral`, `TemplateElement` | Type-based dispatch, empty segment inference |
| `tag` form | `TaggedTemplateExpression` | Tag expression + template |
| `spread` in arrays | `SpreadElement` | Inside `array` form |
| `spread` in objects | `SpreadElement` | Inside `object` form |
| `spread` in calls | `SpreadElement` | Inside function calls |
| `default` in params | `AssignmentPattern` | In function parameter lists |

**Estimated effort**: ~1 day

#### Phase 4 — Patterns

Depends on Phases 2 + 3 (`default`, `alias`, function params, `for-of`).

| Work item | ESTree nodes | Notes |
|-----------|-------------|-------|
| `object` in pattern position | `ObjectPattern` | Context-detect: left of `const`/`let`/`var`/`=`, or in param list |
| `array` in pattern position | `ArrayPattern` | Same context detection |
| `alias` in patterns | `AssignmentProperty` | Reuse from DD-04, two-arg rename, three-arg rename+default |
| `default` in patterns | `AssignmentPattern` | Same node as param defaults |
| `rest` | `RestElement` | In `object` and `array` patterns, must be last |
| `_` skip | `null` in ArrayPattern elements | Array patterns only |
| Nested patterns | Nested `ObjectPattern`/`ArrayPattern` | `alias` target can be a pattern |
| Destructuring in assignment | `AssignmentExpression` with pattern | `=` with `object`/`array` on left |

**Estimated effort**: ~2 days

#### Phase 5 — Classes

Depends on everything: colon syntax (`this:prop`, `super:method`), function conventions, `async` wrapper, destructuring in params, `spread`/`rest`, `default`, `field`.

| Work item | ESTree nodes | Notes |
|-----------|-------------|-------|
| `class` declaration | `ClassDeclaration`, `ClassBody` | Name + superclass list + body |
| `class-expr` | `ClassExpression` | Anonymous, no name |
| Method definitions | `MethodDefinition` | Bare lists in class body |
| `constructor` | `MethodDefinition` (kind: constructor) | Regular method name |
| `get` / `set` markers | `MethodDefinition` (kind: get/set) | Keyword at start of class body member |
| `static` wrapper | `MethodDefinition`/`PropertyDefinition` (static: true) | Wraps any class body member |
| `field` | `PropertyDefinition` | Explicit marker, distinguishes from methods |
| Private via `-` prefix | `PrivateIdentifier` | `-name` → `#_name` in class body context |
| `async` methods | `MethodDefinition` (async: true) | `async` wrapper composes with methods |
| `super` calls | `Super` + `CallExpression` | Already works from Phase 1 |
| `export` + class | `ExportNamedDeclaration` | `export` wraps `class`, already works |

**Estimated effort**: ~2-3 days

#### Phase 6 — Browser shim

Depends on compiler being feature-complete (Phases 1-5).

| Work item | Notes |
|-----------|-------|
| Bundle compiler as single file | reader.js + compiler.js + astring → one distributable |
| `<script type="text/lykn">` scanner | `querySelectorAll`, extract `textContent` |
| Compile-then-eval loop | Compile source to JS string, `eval()` result |
| `src` attribute support | Fetch external `.lykn` files, compile, eval |
| Error handling | Display compile errors in browser console |

**Estimated effort**: ~1 day

### Testing strategy

**Decision**: Both unit tests per form and integration tests that compile complete programs.

**Unit tests**: Each form gets its own test file (`test/forms/import.test.js`, `test/forms/class.test.js`, etc.). Tests compile a lykn snippet and assert the JS output string matches expected output.

**Integration tests**: Complete programs that compile and execute:

| Test program | Exercises |
|-------------|-----------|
| HTTP server | imports, classes, async/await, destructuring, template literals, colon syntax, `new`, arrows |
| CLI tool | imports, function declarations, `for-of`, template literals, `try`/`catch` |
| Utility module | exports, function declarations, destructuring, spread/rest, `?` ternary |
| Browser page | `<script type="text/lykn">`, DOM access via colon syntax, template literals |

### "Done" definition

**Decision**: v0.1.0 is done when all of the following are true:

1. **All unit tests pass** — every form from Phases 1-5 has test coverage.

2. **The reference program compiles and runs:**

```lisp
(import "node:http" (create-server))

(class Handler ()
  (field -count 0)

  (async (-fetch-data (url)
    (const response (await (fetch url)))
    (return (await (response:json)))))

  (async (handle (req res)
    (++ this:-count)
    (const data (await (this:-fetch-data
      "https://api.example.com/data")))
    (const (object name (default age 0)) data)
    (const body (template "Hello, " name "! Age: " age))
    (res:write-head 200 (object (content-type "text/plain")))
    (res:end body))))

(const handler (new Handler))

(const server (create-server (=> (req res)
  (handler:handle req res))))

(server:listen 3000 (=> ()
  (console:log "Server running on port 3000")))
```

3. **The browser shim works** — a simple HTML page with `<script type="text/lykn">` compiles and executes in a browser.

4. **The compiled JS output is clean and readable** — looks hand-written, no runtime dependencies, no artifacts of the compilation process.

### Browser shim: compile-then-eval

**Decision**: The browser shim is included in v0.1.0. It bundles the lykn compiler (reader + compiler + astring) as a single JS file and provides `<script type="text/lykn">` support via compile-then-eval. No interpreter is needed — the compiler runs in the browser, producing a JS string that is eval'd.

**Rationale**: The compiler is tiny (~400 lines + astring at ~16KB). Bundling it for browser use is trivial. This makes lykn immediately usable for experimentation without any build toolchain — just drop a `<script>` tag in an HTML file. The compile-then-eval pattern is proven by BiwaScheme and Wisp.

## Rejected Alternatives

### "Feature complete for modern JS" as v0.1.0 bar

**What**: Include generators, optional chaining, and all ES2022+ features in v0.1.0.

**Why rejected**: Scope creep. v0.1.0 should prove the language is usable, not that it's complete. The deferred features are genuinely less common — you can write real programs without generators or optional chaining.

### Browser shim deferred to v0.1.1

**What**: Ship v0.1.0 without browser support, add the shim in a fast follow.

**Why rejected**: The shim is ~1 day of work and makes lykn dramatically more accessible. It enables experimentation without any toolchain setup. Including it in v0.1.0 makes the release more compelling.

### Interpreter-based browser shim

**What**: Build an s-expression interpreter that runs in the browser.

**Why rejected**: Unnecessary complexity. The lykn compiler already runs in JS. Compile-then-eval produces native JS execution with zero overhead. An interpreter would be slower, more complex, and would need its own testing.

### Reader changes for v0.1.0

**What**: Add reader support for `{}`, `[]`, backticks, `#`, or other non-paren syntax.

**Why rejected**: Every decision from DD-01 through DD-08 was specifically designed to avoid reader changes. The reader stays minimal (four node types: atom, string, list, number). Reader changes are deferred to the future macro system.

## Edge Cases

| Case | Behavior | Example |
|------|----------|---------|
| `.` form still used | Compile-time error or deprecation warning | `(. obj prop)` → error, use colon syntax or `get` |
| `lambda` unchanged | Still works as in v0.0.1 | `(lambda (a b) ...)` → anonymous `FunctionExpression` |
| `block` unchanged | Still works as in v0.0.1 | `(block stmt1 stmt2)` → `BlockStatement` |
| v0.0.1 programs | Should still compile (minus `.` form removal) | Backwards compatible except `.` |
