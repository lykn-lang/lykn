# DD-18.1: Method Calls in Threading Macros

**Status**: Proposed
**Date**: 2026-04-05
**Amends**: DD-18 (Threading Macros and Conditional Binding)

## Problem

DD-18 shows this threading example:

```lisp
(-> user
  (get :name)
  (str:toUpperCase)
  (str:slice 0 10))
```

Expected JavaScript:

```js
user.name.toUpperCase().slice(0, 10)
```

But `(str:toUpperCase)` as a threading step expands to the kernel form
`(str:toUpperCase acc)`, which the JS compiler interprets as
`str.toUpperCase(acc)` — calling a method on the object `str`, not on
the threaded value. This produces:

```js
str.slice(str.toUpperCase(user["name"]), 0, 10)
```

This is wrong. The bug affects all four threading macros (`->`, `->>`,
`some->`, `some->>`) in **both** the JS and Rust implementations.

### Root cause

The colon syntax `a:b` always means "member access on `a`" → `a.b`.
When `str:toUpperCase` appears as a function head, it becomes
`str.toUpperCase(...)`. There is no mechanism in the current kernel to
express "call method `toUpperCase` on the first argument" using
colon-syntax in head position.

Threading macros currently treat every step the same way: insert the
accumulated value as an argument. But method calls need the accumulated
value as the **receiver** (the object), not an argument.

## Decision: keyword-headed steps for method calls

A threading step whose head is a **keyword** (`:name`) is a method call
on the threaded value. The keyword names the method. Additional elements
in the step are arguments to the method.

```lisp
;; Method calls — keyword in head position
(-> user
  (get :name)
  (:to-upper-case)           ;; threadedValue.toUpperCase()
  (:slice 0 10))             ;; threadedValue.slice(0, 10)
```

```js
user["name"].toUpperCase().slice(0, 10)
```

```lisp
;; Function calls — atom in head position (unchanged)
(-> 5
  (Math:floor)               ;; Math.floor(5)
  (+ 3))                     ;; Math.floor(5) + 3
```

```js
Math.floor(5) + 3;
```

### Rules

| Step form | Meaning | Kernel expansion |
|-----------|---------|-----------------|
| `(:method)` | `acc.method()` | `(. acc method)` |
| `(:method a b)` | `acc.method(a, b)` | `(. acc method a b)` |
| `(func arg)` | `func(acc, arg)` for `->` | `(func acc arg)` |
| `bare-sym` | `sym(acc)` | `(sym acc)` |

Keywords undergo the same `to-upper-case` → `toUpperCase` camelCase
conversion as all identifiers.

### New kernel form: `(. obj method args...)`

A new kernel form that compiles to a method call:

```lisp
(. obj method)          → obj.method()
(. obj method a b)      → obj.method(a, b)
(. obj to-upper-case)   → obj.toUpperCase()
```

This form is the compilation target for keyword-headed threading steps.
It can also be used directly outside threading for explicit method calls.

ESTree output: `CallExpression` with a `MemberExpression` callee where
the object is the compiled first argument and the property is an
`Identifier` from the camelCased method name.

### Bare keyword steps

A bare keyword (not wrapped in a list) is also a method call with no
arguments:

```lisp
(-> "hello" :to-upper-case)
;; equivalent to:
(-> "hello" (:to-upper-case))
;; compiles to: "hello".toUpperCase()
```

### Interaction with `get`

`(get obj :key)` remains the way to access a property as a value (not
call it as a method). The keyword-in-threading-step syntax specifically
means "call as method."

```lisp
(-> user (get :name))     ;; user["name"]     — property access
(-> user (:toString))     ;; user.toString()  — method call
```

### What about `some->` and `some->>`?

The same keyword-method syntax applies. Each step in `some->` / `some->>`
that has a keyword head becomes a method call on the nil-checked
intermediate value:

```lisp
(some-> user
  (get :address)
  (get :city)
  (:to-upper-case))
```

```js
(() => {
  const t0 = user;
  if (t0 == null) return t0;
  const t1 = t0["address"];
  if (t1 == null) return t1;
  const t2 = t1["city"];
  if (t2 == null) return t2;
  return t2.toUpperCase();
})()
```

## Rationale

- **Unambiguous**: Keywords are currently invalid as threading step heads,
  so this is purely additive — no breaking changes
- **Consistent**: Keywords already mean "string/name" throughout lykn
  (`:name` → `"name"` in objects, `:key` in `get`). Using them to name
  methods follows the same pattern
- **Explicit**: No guessing about whether `str:toUpperCase` is a
  namespace call or a method call. The user declares intent with syntax
- **Simple**: The kernel form `(. obj method args...)` is a thin mapping
  to JS's `obj.method(args...)` — no complexity

## DD-18 errata

The following DD-18 examples use the old `(str:toUpperCase)` syntax,
which actually performs a namespace call. They should be updated to use
keyword-headed steps:

**Section "-> thread-first"**, lines 50-58:

```lisp
;; OLD (broken — calls str.toUpperCase(acc))
(-> user (get :name) (str:toUpperCase) (str:slice 0 10))

;; NEW (correct — calls acc.toUpperCase().slice(0, 10))
(-> user (get :name) (:to-upper-case) (:slice 0 10))
```

**Section "some-> nil-safe thread-first"**:

```lisp
;; OLD
(some-> user (get :address) (get :city) (str:toUpperCase))

;; NEW
(some-> user (get :address) (get :city) (:to-upper-case))
```

## Implementation

### Phase 1: Add `(. obj method args...)` kernel form

**File**: `src/compiler.js`

Add a new macro entry for `.` that compiles to a `CallExpression` with
a `MemberExpression` callee:

```js
'.'(node, compile) {
  // (. obj method arg1 arg2 ...)
  const obj = compile(node.values[1]);
  const method = node.values[2]; // atom → camelCase
  const args = node.values.slice(3).map(compile);
  return {
    type: 'CallExpression',
    callee: {
      type: 'MemberExpression',
      object: obj,
      property: { type: 'Identifier', name: toCamelCase(method.value) },
      computed: false,
    },
    arguments: args,
  };
}
```

Add tests in `test/forms/` for the new form.

### Phase 2: Update JS threading macros

**File**: `src/surface.js`

In each threading macro (`->`, `->>`, `some->`, `some->>`), detect
keyword-headed steps:

```js
// In the step processing loop:
if (isArray(step)) {
  const head = step.values[0];
  if (isKeyword(head)) {
    // Method call: (:method arg1 arg2)
    // Emit: (. acc methodName arg1 arg2)
    const methodName = atom(head.value); // keyword value as atom
    const args = step.values.slice(1);
    callExpr = array(atom('.'), acc, methodName, ...args);
  } else {
    // Function call: existing behavior
    ...
  }
} else if (isKeyword(step)) {
  // Bare keyword: :method → (. acc method)
  callExpr = array(atom('.'), acc, atom(step.value));
} else {
  // Bare symbol: existing behavior
  ...
}
```

Update threading tests in `test/surface/threading.test.js` and
`test/surface/some-threading.test.js`.

### Phase 3: Update Rust emitter

**File**: `crates/lykn-lang/src/emitter/forms.rs`

Update `apply_threading_step` to detect keyword-headed steps:

```rust
fn apply_threading_step(acc: SExpr, step: &ThreadingStep, first: bool, ...) -> SExpr {
    match step {
        ThreadingStep::Call(exprs) => {
            let head = &exprs[0];
            if let SExpr::Keyword { value, .. } = head {
                // Method call: emit (. acc methodName args...)
                let method = atom(value);
                let rest: Vec<SExpr> = exprs[1..].iter().map(|e| emit_expr(e, ctx, registry)).collect();
                let mut items = vec![atom("."), acc, method];
                items.extend(rest);
                list(items)
            } else if first {
                // Thread-first function call (existing)
                ...
            } else {
                // Thread-last function call (existing)
                ...
            }
        }
        ThreadingStep::Bare(expr) => {
            if let SExpr::Keyword { value, .. } = expr {
                // Bare keyword method call: emit (. acc methodName)
                list(vec![atom("."), acc, atom(value)])
            } else {
                // Bare symbol function call (existing)
                list(vec![emit_expr(expr, ctx, registry), acc])
            }
        }
    }
}
```

## Verification

```sh
# JS pipeline
deno eval "import {lykn} from './src/index.js'; \
  console.log(lykn('(-> \"hello\" (:to-upper-case))'))"
# → "hello".toUpperCase();

deno eval "import {lykn} from './src/index.js'; \
  console.log(lykn('(-> user (get :name) (:to-upper-case) (:slice 0 10))'))"
# → user["name"].toUpperCase().slice(0, 10);

# Rust pipeline
echo '(-> "hello" (:to-upper-case))' > test_tmp.lykn
./target/release/lykn compile test_tmp.lykn
# → "hello".toUpperCase();

# Nil-safe threading
deno eval "import {lykn} from './src/index.js'; \
  console.log(lykn('(some-> user (get :name) (:to-upper-case))'))"
# → IIFE with t.toUpperCase() at the end

# Function calls unchanged
deno eval "import {lykn} from './src/index.js'; \
  console.log(lykn('(-> 5 (Math:floor) (+ 3))'))"
# → Math.floor(5) + 3;

# Full test suites
deno task test
cargo test
```
