# lykn v0.3.0 — Implementation Plan

**Generated from**: DD-15 through DD-21
**Date**: 2026-03-28
**Goal**: lykn/surface language — functional, typed, safe, with a Rust
toolchain library and parallel JS reference implementation

---

## Architecture

```
.lykn source
    │
    ▼
Rust toolchain library
    ├── Reader (s-expr tree + source locations)
    ├── Classifier (typed surface AST)
    ├── Macro Expander (user macros via Deno subprocess)
    ├── Collection (type registry, signatures, prelude)
    ├── Analysis (exhaustiveness, overlap, scope, unused bindings)
    ├── Kernel Emitter (surface → kernel AST as JSON)
    └── Diagnostics (errors, warnings, s-expr serializer)
    │
    ▼
Kernel AST (JSON — flat arrays matching JS reader format)
    │
    ▼
JS kernel compiler (reader.js + compiler.js + astring)
    │
    ▼
JavaScript output
```

**Parallel path** (JS surface compiler):

```
.lykn source
    │
    ▼
JS surface compiler (all surface forms as macros, DD-13 pipeline)
    │
    ▼
Kernel AST (JSON — canonical test fixtures)
    │
    ▼
JS kernel compiler → JavaScript output
```

Both paths must produce identical kernel JSON for the same input.

---

## Development Strategy

**JS-first, Rust follows**: Implement surface form expansion in JS first
(the macro infrastructure from v0.2.0 exists). Capture kernel JSON output
as test fixtures. Build the Rust implementation to match. The JS path is
faster to iterate; the Rust path provides static analysis.

**Two workstreams** can proceed in parallel:

- **Workstream A (JS)**: Surface form macros in JS. No static analysis.
  Produces canonical test fixtures.
- **Workstream B (Rust)**: Toolchain library modules. Reader, classifier,
  analysis, emitter. Uses Workstream A output as test oracle.

Workstream A should lead Workstream B by at least one phase — the JS
implementation for each form should be working before the Rust
implementation starts.

## Phase 1 — JS Surface Form Macros (Foundation)

**Dependencies**: v0.2.0 macro system (DD-10–14) must be working.
**Workstream**: A (JS)
**Estimated effort**: ~3–4 days

Implement the simplest surface forms as macros in the JS expander. These
are the forms that require no static analysis — pure syntactic transforms.

### 1.1 `bind` macro

`(bind name value)` → `(const name value)`
`(bind :type name value)` → `(const name value)` (type annotation
recorded for future use, not emitted in JS-only path)

### 1.2 `obj` macro

`(obj :name "Duncan" :age 42)` → `(object (name "Duncan") (age 42))`

Transform keyword-value alternation to grouped pairs.

### 1.3 Cell vocabulary macros

`(cell value)` → `(object (value value))`
`(express c)` → `c:value`
`(swap! c f)` → `(= c:value (f c:value))`
`(swap! c f arg1 arg2)` → `(= c:value (f c:value arg1 arg2))`
`(reset! c v)` → `(= c:value v)`

### 1.4 Threading macros (`->`, `->>`)

Pure syntactic rewriting. No nil checks, no IIFE.

`(-> x f g)` → `(g (f x))`
`(-> x (f a) (g b))` → `(g (f x a) b)`
`(->> x (f a) (g b))` → `(g a (f b x))`

### 1.5 Immutable update macros

`(assoc obj :key val)` → `(object (spread obj) (key val))`
`(dissoc obj :key)` → appropriate spread + destructuring
`(conj arr val)` → `(array (spread arr) val)`

### Phase 1 output

Test fixtures: JSON kernel AST for each form. These become the test
oracle for Workstream B.

## Phase 2 — JS Surface Form Macros (Complex Forms)

**Dependencies**: Phase 1 complete.
**Workstream**: A (JS)
**Estimated effort**: ~5–7 days

Implement the forms that require IIFE codegen and more complex
transformation.

### 2.1 `type` macro

`(type Option (Some :any value) None)` expands to:

```lisp
(function Some (value) (return (object (tag "Some") (value value))))
(const None (object (tag "None")))
```

Dev-mode type checks emitted inside constructor functions:

```lisp
(function Some (value)
  (if (condition) (throw (new TypeError "...")))
  (return (object (tag "Some") (value value))))
```

### 2.2 `func` macro (single clause)

`(func name :args (:type1 a :type2 b) :returns :type :pre expr :post expr :body body...)` expands to `(function name ...)` with type checks, contract checks, and `_result` binding for `:post`.

### 2.3 `func` macro (multi-clause)

Multi-clause `func` expands to rest params + dispatch chain:

```lisp
(function name (...args)
  (if (dispatch-check-1) (body-1))
  (if (dispatch-check-2) (body-2))
  (throw (new TypeError "no matching clause")))
```

### 2.4 `match` macro

Statement position: if-chain.
Value position: IIFE.
Tail position of func: if-chain with return.

Pattern compilation: ADT constructor patterns check `.tag`, literals
check `===`, structural patterns check `typeof` + `in` + `===`.

### 2.5 `some->` / `some->>` macros

IIFE with sequential `const` bindings and `== null` checks after each
step.

### 2.6 `if-let` / `when-let` macros

Pattern-based conditional binding. Value position: IIFE. Statement
position: const + if.

### 2.7 `fn` / `lambda` macros

`(fn (:type x :type y) body)` → `(=> (x y) body)` with dev-mode type
checks.

### Phase 2 output

Complete test fixture set: every surface form with multiple variations
(typed, untyped-with-any, multi-clause, nested patterns, guards,
threading, contracts). JSON kernel AST for each.

## Phase 3 — Rust Toolchain Library (Foundation)

**Dependencies**: None (can start in parallel with Phase 1).
**Workstream**: B (Rust)
**Estimated effort**: ~5–7 days

Build the foundational Rust modules that all tools will use.

### 3.1 Reader module

Implement the lykn reader in Rust following DD-01 and DD-12:

- S-expression parser: atoms, lists, strings, numbers, booleans
- Colon syntax: splitting, leading `:` → keyword type
- `#` dispatch: `#a(...)`, `#o(...)`, `#;`, `#NNr`, `#|...|#`
- Source locations: file, line, column, span on every node
- Output: generic s-expression tree (untyped)

**Test**: Parse the same `.lykn` files the JS reader parses, verify
identical structure (modulo source location format).

### 3.2 Surface AST module

Define the typed AST enum with variants for all surface forms:

`Func`, `Bind`, `Match`, `Type`, `Obj`, `Cell`, `Express`, `Swap`,
`Reset`, `ThreadFirst`, `ThreadLast`, `SomeThreadFirst`,
`SomeThreadLast`, `IfLet`, `WhenLet`, `Fn`, `Lambda`, `Macro`,
`ImportMacros`, `KernelPassthrough`

Each variant carries structured fields (e.g., `Func` has `name`,
`clauses`, each clause has `args`, `returns`, `pre`, `post`, `body`).

### 3.3 Classifier module

Transform generic s-expression tree → typed surface AST:

- Dispatch on head symbol of each list form
- Validate surface form structure (reject malformed forms with
  source-located errors)
- Classify unknown head symbols as function calls
- Classify kernel form names as `KernelPassthrough`

**Test**: Classify the test fixtures from Phase 1/2, verify correct
AST node types.

### 3.4 Diagnostics module

- Structured diagnostic type: severity, message, location, source form
- S-expression serializer: AST → source text (for contract error messages)
- Enhanced message templates for blessed types
- Output formats: text (human-readable), JSON (tooling)

## Phase 4 — Rust Toolchain Library (Analysis)

**Dependencies**: Phase 3 complete. Phase 2 test fixtures available.
**Workstream**: B (Rust)
**Estimated effort**: ~7–10 days

Build the analysis passes. This is the most complex phase.

### 4.1 Type registry

- Data structure: type name → constructors → fields → field types
- Blessed type identification by module path
- Constructor lookup by name → owning type
- Population from `Type` AST nodes during collection phase

### 4.2 Prelude injection

- Auto-import `Option`, `Some`, `None`, `Result`, `Ok`, `Err` from
  `lykn/core`
- Inject before classification
- Shadowing allowed — local definitions win, lose compiler enhancement

### 4.3 Scope tracker

- Binding introduction for all forms (surface + kernel)
- Usage tracking (symbol references → mark binding as used)
- Shadowing detection (same name in nested scope → warning)
- Unused binding detection (introduced but never used → warning)
- `_` prefix exempt, exports exempt, constructor names exempt

### 4.4 Maranget's exhaustiveness algorithm

Core implementation:

- Deconstructed pattern representation (separate from surface AST)
- Pattern matrix construction from match clauses
- Recursive specialization by head constructor
- Exhaustiveness check: are all inputs covered?
- Usefulness check: is this clause reachable?
- Guard handling: guarded clauses excluded from coverage set

### 4.5 `match` exhaustiveness checking

Wire Maranget to `Match` AST nodes:

- Pattern-based type inference: look up constructors in type registry,
  verify all belong to same type
- Mixed constructor type error (constructors from different types)
- Mixed constructor + literal type error
- Literal-only requires `_`
- Structural requires `_`
- Report uncovered variants in error messages
- Enhanced messages for blessed types

### 4.6 `func` overlap detection

Wire Maranget to multi-clause `Func` AST nodes:

- `:args` patterns as rows in pattern matrix
- Type keywords as constructors from built-in type set
- Arity as column count
- Overlap = two clauses both useful for some input
- Report overlapping clause pairs in error messages

### Phase 4 output

Analysis module passes all test cases. Exhaustiveness errors caught
for non-exhaustive matches. Overlap errors caught for conflicting
func clauses. Unused bindings flagged.

## Phase 5 — Rust Toolchain Library (Emission)

**Dependencies**: Phase 4 complete.
**Workstream**: B (Rust)
**Estimated effort**: ~5–7 days

Build the kernel emitter. Transform typed surface AST → kernel JSON.

### 5.1 Expression context tracking

- Three contexts: `Statement`, `Value`, `Tail`
- Propagation rules: `Bind` value → `Value`, `Func` last body expr with
  `:returns` → `Tail`, top-level → `Statement`
- IIFE emission in `Value` context
- If-chain with `return` in `Tail` context

### 5.2 Surface form emission rules

Implement emission for each surface AST node type:

- `Bind` → `const`
- `Func` → `function` with type checks, contracts, dispatch
- `Match` → if-chain or IIFE depending on context
- `Type` → constructor functions + const declarations
- `Obj` → `object` with grouped pairs
- Cell forms → property access / assignment
- Threading → nested calls or IIFE with nil checks
- `IfLet`/`WhenLet` → pattern check + IIFE or statement
- `Fn`/`Lambda` → arrow function or function expression
- `KernelPassthrough` → raw s-expression unchanged

### 5.3 Type check and contract emission

- Type check patterns: `typeof`, `Array.isArray`, NaN exclusion,
  tag checks for user-defined types
- Contract check pattern: negated condition + `throw new Error`
- S-expression serialization for contract error messages
- `--strip-assertions` flag: skip all assertion emission

### 5.4 `await` handling

- Detect `await` in target position → hoist to `const` before IIFE
- Detect `await` in branch body → async IIFE + outer `await`
- Verify enclosing function is `async` when async IIFE is emitted

### 5.5 JSON serialization

- Kernel AST → JSON matching JS reader flat-array format
- Symbols as strings, keywords as their string values
- Nested arrays for nested forms
- Output to file or stdout

### Phase 5 output

Rust compiler produces kernel JSON. Compare against JS surface
compiler output (Phase 2 test fixtures). Must match exactly.

## Phase 6 — Macro Expander (Rust)

**Dependencies**: Phase 3 (reader, classifier). Phase 2 (JS macros
working for reference).
**Workstream**: B (Rust)
**Estimated effort**: ~4–5 days

### 6.1 Deno subprocess protocol

- JSON schema for macro invocation: macro body (JS source), arguments
  (generic s-expression tree as JSON), macro environment API availability
- Invoke Deno with `deno eval` or similar
- Parse expanded AST from subprocess stdout (JSON)
- Error handling: subprocess failure → compile error with stderr

### 6.2 Three-pass pipeline (Rust)

Implement DD-13's pipeline logic in Rust:

- Pass 0: process `import-macros`, load macro module files
- Pass 1: compile `macro` definitions (iterative fixed-point)
- Pass 2: expand macro invocations (recursive top-down, per-node limit)

Built-in surface forms are NOT processed by this pipeline — they're
already classified as typed AST nodes.

### 6.3 Re-classification after expansion

- Expanded forms (generic s-expression trees from Deno subprocess)
  feed back through the classifier
- Expanded surface forms become typed AST nodes
- Expanded kernel forms become `KernelPassthrough`

### Phase 6 output

User-defined macros expand correctly in Rust. Output matches JS
expander output for the same macro definitions and invocations.

## Phase 7 — Integration and CLI

**Dependencies**: Phases 1–6 complete.
**Estimated effort**: ~3–4 days

### 7.1 `lykn compile` command

- Parse CLI args: input files, `--strip-assertions` flag
- Read `.lykn` source → Rust pipeline → kernel JSON → JS kernel compiler
  → JavaScript output
- Error reporting: diagnostics to stderr, output to stdout or file

### 7.2 End-to-end integration tests

Write complete `.lykn` programs using surface syntax and verify they
compile and run:

**Test 1: Data processing with ADTs**

```lisp
(type Shape
  (Circle :number radius)
  (Rect :number width :number height)
  (Point))

(func area
  :args (:Shape s)
  :returns :number
  :body
  (match s
    ((Circle r) (* Math:PI (* r r)))
    ((Rect w h) (* w h))
    (Point 0)))

(bind shapes (array (Circle 5) (Rect 3 4) Point))
(bind total (->> shapes
  (map area)
  (reduce + 0)))
(console:log (template "Total area: " total))
```

**Test 2: Option handling with nil-safe threading**

```lisp
(func find-user
  :args (:string id)
  :returns :Option
  :body
  (if (= id "1")
    (Some (obj :name "Duncan" :email "d@example.com"))
    None))

(bind greeting (some-> (find-user "1")
  (get :name)
  (str:toUpperCase)))
(console:log (if-let (name greeting)
  (template "Hello, " name "!")
  "User not found"))
```

**Test 3: Contracted function with multi-clause dispatch**

```lisp
(func safe-divide
  (:args (:number a :number b)
   :returns :number
   :pre (not= b 0)
   :body (/ a b))

  (:args (:number a)
   :returns :number
   :body (/ 1 a)))

(console:log (safe-divide 10 3))
(console:log (safe-divide 5))
```

**Test 4: Cells and mutation**

```lisp
(bind counter (cell 0))
(func inc-counter
  :args ()
  :body (swap! counter (fn (:number n) (+ n 1))))

(inc-counter)
(inc-counter)
(inc-counter)
(console:log (template "Count: " (express counter)))
```

### 7.3 JS surface compiler as test oracle

- Run each integration test through both the JS surface compiler
  (Workstream A) and the Rust surface compiler (Workstream B)
- Diff the kernel JSON output
- Any difference is a bug in one or the other
